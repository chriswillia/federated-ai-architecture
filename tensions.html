Trade-offs & references

A federated AI environment is the right direction, but it isn’t free of trade-offs. Below: five contradictions worth naming up front before they surface in delivery, and the primary sources behind the architecture.

Five trade-offs worth naming up front

Name these before delivery does. It builds credibility and shortens the architecture debate inside your own organization.

Single control plane vs vendor reality
Every hyperscaler ships its own control surface. “One control plane” is an outcome — it requires integration glue, not just a logo.
Workload identity is harder than it sounds
Human SSO is solved. Federating identities for agents — across Entra, AWS IAM, GCP IAM, and SaaS — is still emerging and operationally fragile.
Citizen development vs governance load
Letting business users build agents accelerates value and multiplies what governance, security, and FinOps must inspect. Plan for the load.
Cost attribution at agent granularity
Token-level showback per agent, per user, per business unit isn’t standard yet. Tags, model routing, and bill-back discipline are still bespoke.
MCP and A2A overlap at the edges
The protocols are complementary in theory (tool calls vs agent collaboration), but their boundaries blur in practice. Expect overlap and version churn.

Seller objection handling

Lead with the trade-off. Do not deny it; clarify the decision and anchor the response in portable standards first.

When the customer says: “One control plane is unrealistic.”

You say: “A control plane is a consistent operating outcome, not a single vendor console. Use common inventory, identity, policy, and telemetry contracts; integrations do the work across platforms. MCP, A2A, OAuth/OIDC, OpenTelemetry, and an API gateway make the architecture portable.”

When the customer says: “Workload identity is too fragile.”

You say: “It is a real implementation challenge. Start with the highest-impact agent paths, use least-privilege workload identity and short-lived credentials, and measure federation failures before expanding. Do not claim that human SSO alone solves agent identity.”

When the customer says: “Citizen development will overwhelm governance.”

You say: “It can, unless the fast path inherits registration, approved tools, DLP, and tiered review. The goal is not to stop builders; it is to make safe defaults cheaper than workarounds.”

When the customer says: “We cannot attribute AI cost accurately.”

You say: “Perfect per-agent bill-back is still immature. Begin with accountable owners, tags, outcome telemetry, and showback. Improve precision as usage and model routing mature rather than waiting for a perfect billing schema.”

When the customer says: “MCP and A2A overlap, so standards are not ready.”

You say: “The boundaries are evolving. Treat MCP as a tool-context pattern and A2A as an agent-collaboration pattern, isolate both behind an API gateway, and version contracts so changing protocols do not reshape every agent.”

When the customer says: “You call this vendor-neutral, but the control plane is always Microsoft.”

You say: “The framework is vendor-neutral when its control points are standards-based and its products are substitutable. Microsoft is an example anchor stack, not a prerequisite: retain the architecture with another identity provider, policy engine, SIEM, catalog, and API gateway. Be explicit where a chosen product has stronger native coverage.”
