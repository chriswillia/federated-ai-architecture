Agentic governance

Turn an unknown agent estate into accountable automation.

Agentic governance is the operating model that discovers agents across your estate, establishes ownership and boundaries, applies controls at runtime, and creates evidence that those controls work.

Discovery is not governance. Finding an agent gives you a signal. Governance begins when the agent has a known owner, identity, purpose, risk tier, permitted tools and data boundaries, and an auditable lifecycle.

The operating principle

Let teams build. Require agents to earn trust.

The objective is not to route every agent through a single runtime. It is to make the controls that matter portable across every runtime.

01 · Know

Discover the whole estate

Combine inventory, cloud, SaaS, endpoint, network, and developer signals. Treat managed agents and shadow usage as different states in one estate view.

02 · Bound

Define accountable autonomy

Record the agent's owner, business purpose, risk tier, identity, tools, data sources, approvals, and stop conditions in a versioned trust contract.

03 · Prove

Enforce and evidence

Apply identity, data policy, tool controls, monitoring, and human oversight at the right boundaries. Preserve the evidence needed to investigate and improve.

Portable control boundaries

Enforce where the agent meets the enterprise.

This creates a vendor-neutral control pattern. Products can change; the boundaries remain.

Identity boundary

Use a distinct workload identity, least-privilege scopes, and delegated access only where needed. An agent should never inherit a human's broad access by default.

Portable pattern: OAuth/OIDC, workload identity, scoped credentials.

Tool and API boundary

Route tool calls through governed interfaces. Allow known tools, validate inputs and outputs, rate-limit actions, and require approval for irreversible operations.

Portable pattern: MCP, API gateway, signed tool catalog, policy-as-code.

Data boundary

Apply data classification, retrieval restrictions, DLP, and retention rules to prompts, context, outputs, and tool calls—not just the source repository.

Portable pattern: data labels, DLP, access policy, governed retrieval.

Runtime boundary

Apply safety, model-routing, execution limits, and stop conditions where the agent reasons and acts. Escalate when confidence, risk, or policy thresholds are crossed.

Portable pattern: guardrails, quotas, human-in-the-loop, kill switch.

Evidence boundary

Capture a coherent trace from request to tool call to outcome. Tie activity to the agent identity, policy decision, data classification, and business owner.

Portable pattern: OpenTelemetry, common event schema, immutable audit records.

Lifecycle boundary

Require review when an agent changes its model, tools, data sources, autonomy, or audience. Remove access when the use case or ownership ends.

Portable pattern: ABOM, change approval, re-certification, decommissioning workflow.
What good looks like

A governance record that supports action.

Inventory

One record per production agent

Owner, sponsor, purpose, runtime, model, tools, data sources, permissions, risk tier, and current lifecycle state.

Trust contract

Machine-enforceable boundaries

Allowed tools, allowed data, approval requirements, logging obligations, escalation conditions, and review cadence.

Operational evidence

Proof the controls operate

Evaluation results, policy decisions, traces, exceptions, incidents, cost signals, and measured business outcomes.

A practical starting point

Pick one real agent, then map its path to trust.

Inventory its identity, tools, data, actions, owner, and evidence. Identify the first unmanaged boundary. Improve that boundary before expanding to the next scenario.
